# Per-Theme Block Colors Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let page-builder editors set a separate custom background color for light vs dark theme on a section block, rendered automatically by the storefront's class-based dark mode, with a Light/Dark toggle in the editor that parallels the vi/en switcher.

**Architecture:** Add a paired `backgroundCustomDark` hex field beside the existing `backgroundCustom` (no Payload localization — theme is not a locale). The storefront emits both colors as CSS custom properties and lets a `.dark`-scoped CSS rule pick. The editor gains a page-level theme-mode toggle (UI state, no data refetch) that (a) toggles `.dark` on the preview iframe via a postMessage and (b) routes the appearance color picker to the light or dark slot.

**Tech Stack:** Next.js 15, Payload CMS 3.84 (Postgres adapter), Tailwind CSS 4 (`darkMode: 'class'`), TypeScript, Vitest + @testing-library/react.

## Global Constraints

- Package manager is pnpm, but `pnpm <script>` fails via runDepsStatusCheck — call binaries in `node_modules/.bin/` directly (e.g. `node_modules/.bin/vitest run ...`, `node_modules/.bin/payload ...`).
- New Payload block/field changes REQUIRE a generated migration or the storefront throws Postgres `42P01` at runtime.
- `DATABASE_URL` is a remote Prisma Postgres whose migration ledger may be behind `src/migrations` — always run `migrate:status` before `migrate:create`.
- Payload relationship writes need native numeric ids (not relevant here, but do not stringify ids).
- Follow existing file conventions: appearance fields live in `src/payload/blocks/_appearance.ts`; appearance→class mapping lives in `lib/page-builder.ts`.
- Test command (single file): `node_modules/.bin/vitest run <path>`. Typecheck: `node_modules/.bin/tsc --noEmit`.

```ts
// lib/page-builder/themed-color.ts — maps a base color field to its per-theme slot.
// A "themed color" stores two hex values: <base> for light, <base>Dark for dark.
// The editor's theme-mode toggle decides which slot the single picker writes to.

export type ThemeMode = 'light' | 'dark';

/** Base appearance color fields that have a paired `<base>Dark` slot. */
export const THEMED_COLOR_BASES: ReadonlySet<string> = new Set(['backgroundCustom']);

/** The dark-slot field names, hidden as standalone fields in the panel. */
export const THEMED_DARK_SLOTS: ReadonlySet<string> = new Set(['backgroundCustomDark']);

/** The field name the picker binds to for the given theme mode. */
export function activeColorSlot(baseName: string, mode: ThemeMode): string {
  return mode === 'dark' ? `${baseName}Dark` : baseName;
}
```
