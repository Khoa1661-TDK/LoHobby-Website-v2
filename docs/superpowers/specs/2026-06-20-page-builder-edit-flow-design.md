# Page Builder Edit Flow — Design

Date: 2026-06-20
Status: Approved (design)

## Problem

The new visual page builder (`/[locale]/build/[slug]`, `EditorShell` + iframe preview)
is only reachable by opening a page in the Payload admin form and clicking
"Open visual builder" — and that button only appears after a page is saved with a
slug. Creating a page cannot go straight into the builder, and the old admin
`layout` blocks editor (with `livePreview`) is still the default edit surface.

We want to make the visual builder the primary editing surface ("retire" the old
admin blocks editor), give a one-click path to create a blank page and edit it in
the builder, and ensure the home/main page always appears in the Pages list and
renders on the storefront.

## Goals

1. A "+ New page" action that creates a blank draft page and redirects straight
   into the visual builder.
2. Clicking a page in the Payload admin Pages list opens the visual builder
   (edit mode), not the old admin blocks form.
3. The home/main page always exists in the Pages list, is editable in the builder
   like any other page, and renders at storefront `/`.
4. The builder is self-sufficient for the common case (title + status/publish),
   with an escape hatch to the raw admin form for slug/SEO.

## Non-Goals

- Rebuilding SEO (`meta`) editing inside the builder. SEO/slug stay in the Payload
  admin form, reached via an "Advanced settings" link.
- Changing the storefront render path (`/` and `/pages/[slug]` already render
  builder layouts).
- Changing block schemas or the iframe preview postMessage protocol.

## Approach

### 1. "+ New page" → blank page → builder

- New API route `POST /api/page-builder/create-page` (admin-gated via
  `isAuthorizedAdmin`, same pattern as `set-homepage`):
  - Creates a `pages` doc: `title: 'Untitled'`, empty `layout: []`, `status: 'draft'`.
    Slug is auto-generated by the existing `autoSlugFromTitle` hook
    (`resolveCollectionSlug` guarantees uniqueness → `untitled`, `untitled-2`, …).
  - Returns `{ href: '/${defaultLocale}/build/${slug}' }`.
- Custom client component mounted in the Pages list via
  `admin.components.beforeListTable`. It POSTs to the route and
  `router.push(href)` on success.

### 2. Click-through to builder (Option A — custom title Cell)

- Add a custom `admin.components.Cell` to the `title` field that renders a link to
  `/${defaultLocale}/build/${slug}` instead of the default link to the admin edit
  view. Uses the page's `slug` from the row data.
- The old `layout` blocks field and the collection `livePreview` config are
  **hidden from the admin form** (`admin.hidden: true` on the `layout` field;
  remove/disable `livePreview`). The `layout` field stays in the schema so the
  builder continues to read/write it; it is simply no longer the edit surface.
  This is the concrete "retirement" of the old builder.
- The Payload admin edit form remains intact and reachable (for slug/SEO) via the
  builder's "Advanced settings" link — nothing is deleted, so there is a safety net.

### 3. Builder self-sufficiency

- `EditorShell` header gains an inline editable **title** field, alongside the
  existing status badge and Publish button.
- Extend `PATCH /api/pages/[id]` to accept an optional `title` (validated as a
  non-empty string). Title autosaves through the same debounced path as layout.
- Add an **"Advanced settings"** link in the builder header to the raw Payload
  admin doc form (`/admin/collections/pages/${id}`) for slug/SEO editing.

### 4. Slug stability after creation

- Change `autoSlugFromTitle` so the slug is generated **only when the slug is
  empty** (i.e. on first create). On update of an existing page, the stored slug
  is preserved even when the title changes.
- Rationale: the builder URL (`/build/[slug]`) and the iframe preview `src` are
  keyed by slug. Regenerating the slug on every title edit would change the URL
  mid-session and break the open builder. Slug changes become an explicit action
  in the Advanced settings form.

### 5. Home auto-seed

- Seed script (e.g. `scripts/seed-home-page.ts`) that ensures a `pages` doc with
  `slug: 'home'` exists, using `buildHomeSeedLayout()` for its layout when created.
  Idempotent: no-op if `home` already exists.
- Result: `home` always appears in the Pages list, is editable in the builder, and
  storefront `/` renders it (already wired in `app/[locale]/(storefront)/page.tsx`
  via `getHomePage()`).
- The existing "Set as homepage" button / `set-homepage` route can remain; its
  create-if-missing logic overlaps with the seed and stays as a manual fallback.

## Components / Files

| File | Change |
|------|--------|
| `app/api/page-builder/create-page/route.ts` | New: admin-gated blank-page create + builder href |
| `src/payload/components/NewPageButton.tsx` | New: list-view button that creates + redirects |
| `src/payload/components/PageTitleCell.tsx` | New: title Cell linking to the builder |
| `src/payload/collections/Pages.ts` | `beforeListTable` button, title `Cell`, hide `layout`, drop `livePreview`, slug-stable hook |
| `components/page-builder/EditorShell.tsx` | Inline title edit + Advanced settings link |
| `components/page-builder/use-autosave.ts` | Include `title` in patch body |
| `app/api/pages/[id]/route.ts` | Accept optional `title` in PATCH |
| `scripts/seed-home-page.ts` | New: idempotent home-page seed |

## Data Flow

1. Admin clicks **+ New page** → `POST /api/page-builder/create-page` →
   blank draft created → redirect to `/build/<slug>` → `EditorShell` loads.
2. Admin clicks a page **title** in the list → link → `/build/<slug>` → builder.
3. In the builder, editing title/layout autosaves via `PATCH /api/pages/<id>`;
   Publish sets `status: 'published'`.
4. Slug/SEO edits happen in the admin form via **Advanced settings**.
5. Storefront `/` renders the `home` page; `/pages/<slug>` renders other pages.

## Error Handling

- `create-page` route returns 401 when not an authorized admin (same as
  `set-homepage`); the button surfaces a non-blocking failure state.
- `PATCH /api/pages/[id]` rejects an empty/blank `title` with 400; the builder
  shows the existing "Save failed — retry" status.
- Seed script is idempotent and logs whether it created or skipped `home`.

## Testing

- Unit: `autoSlugFromTitle` preserves an existing slug on title-change update, and
  generates a slug only when empty (reproduces the slug-stability requirement).
- Unit: `create-page` route creates a draft with empty layout and returns the
  correct builder href; returns 401 unauthorized.
- Unit: `PATCH /api/pages/[id]` accepts a valid title and rejects an empty one.
- Unit/integration: home seed script is idempotent (creates once, skips when
  present).

## Trade-offs

- SEO/slug editing stays in the admin form rather than the builder — accepted to
  keep scope contained and avoid rebuilding the SEO UI.
- Hiding `layout` from the admin form means the only layout editor is the builder;
  the admin form is no longer a fallback for block editing (intended).
- Slug becomes stable after creation; renaming no longer changes the URL, which is
  the desired behavior but differs from the previous auto-regenerating slug.
